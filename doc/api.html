Introduction

The API to Mobile Monitor for OTCS (MOTCS_API) defines all CURD operation available for data in the system. The API uses one entry point defined by the reverse proxy. Recommended resource name is '/motcs/rest'. The resource takes two mandatory parameters and one optional. One is cmd and the second one data. The last parameter is called processSupportData and is used by JOSNP calls only. The cmd stands for command and is always plain string formated. The second parameter stands for payload data and is always JSON serialized. The payload should be always URI-protected / URI-encoded to guarantee the transfer. The last parameter, the optional one, is importatn for JSONP integration. User can send over the name of the callback routine that is supposed to be called back on client. The return data are then serialized as a parameter of the callback method. The API supports both GET and POST method. We strongly recommend to use POST method exclusively to avoid any problem with data size being transferred.

The following section described all available command that the API accepts. The index is given for commands only. Remember that the complete API call must contain the complete URI specification and parameter section to specify the URI seceltion procedure.

Example


	http://m.opentext.net/motcs/rest?cmd=insert_case&data={"caseNo":"192","caseSubject":"Hello World"}

	http://m.opentext.net/motcs/rest?cmd=insert_case&data={"caseNo":"192","caseSubject":"Hello World"}&processSupportData=Ext.JsonP.callback19

API Specification

API description does not use the optional parameters. They are sent authomatically by the Sencha Touch framework when configured for JSONP data transfer.

Data Write Methods

insert_case

POST

/motcs/rest?cmd=insert_case&data={parameters}

Inserts new case into relational database. The command takes two parameters only. The additional metadata are pre-set to default value and can be modified by other API calls.

request query parameters
parameter value description

caseNo

integer

Case number originating in ITSM database.

caseSubject

string

Subject of the incidence case. The subject corresponds with the synopsis of the problem reported by the customer.
Example

{
	"caseNo":"111888",
	"caseSubject":"OTCS Ticket 111999: Problem saving annotation in WinViewer 10.0.0. Update 006"
}
available response representations:
  • 200 - OK [expand]

    application/json (property)

    Example
    {
  "support_data":{ 
    "feed": {
      "title":"support data", 
      "entries":{
        "code":229,
        "message":""
      }
    }
  },
  "responseDetails":null,
  "responseStatus":200
}

    Returned if the insert operation succeeded. The returned code brings the ordinal number of inserted case in the database (id_01 from the table t01_case).

  • 404 - Duplicate ID [expand]

    application/json (property)

    Example
    {
  "support_data":{ 
    "feed": {
      "title":"support data", 
      "entries":{
        "code":404,
        "message":"This case is already inserted"
      }
    }
  },
  "responseDetails":null,
  "responseStatus":404
}

    Returned if the insert operation fails. The database already contains the case referred under the specified caseNo parameter. The caseNo must be unique.

  • 503 - Service Temporarily Unavailable [expand]

    The application server or reverse proxy are not available. One of them or both might crash.

mdb_insert_mail

POST

/motcs/rest?cmd=mdb_insert_mail&data={parameters}

Inserts new mail into database. The link to the case is the case number. The API does not verify whether the case already exist. One can insert new e-mail having a case number that does not exist in the system yet. In that case the stored e-mail cannot be retrieved. It is not linked with any case. The system stores e-mail in different database than the relational data about the cases. Therefore the API does not implement any constraint between e-mail database and relation database.

request query parameters
parameter value description

caseNo

integer

Case number originating in ITSM database.

mailSubject

string

Subject of the e-mail.

mailSender

string

String representation of the sender. Is not SMTP address, is the Outlook string.

mailHTMLBody

string

HTML formatted body of the e-mail.

sentOn

string

Create (sent) date in string representation. The Outlook exposes the datetime data type as Variant via the OLE interface. That representation is illegible to JavaScript and therefore the function 'Date()', 'Time()' respectively are called on the Variant before sending to JavaScript layer. The pattern "dd.m.yyyy hh:mm:ss <AM|PM>". (e.g.: "12.3.2013 12:21:35 PM") is resulting.

attachements

string

List of attachments. It is semicolon separated list, delivered by outlook. The list is either epmty or starts with the sequence '; ' followed by name of attachment(s). The attachments are semicolon delimited.
Example

{
	"caseNo":"165445",
	"mailSubject":"OTCS Ticket 165445: Images distorted in full view.",
	"mailHTMLBody":"<HTML><BODY><P>Hello Scan Team, the customer report image distortion</P></BODY></HTML>",
	"sentOn":"08.3.2013 11:37:21 AM",
	"mailSender":"Open Text Customer Support",
	"attachments":"; ixclient.log"
}
available response representations:
  • 200 - OK [expand]

    The response body contain 'application/json' (property)

    Example
    {
  "support_data":{ 
    "feed": {
      "title":"support data", 
      "entries":{
        "code":200,
        "message":"OK"
      }
    }
  },
  "responseDetails":null,
  "responseStatus":200
}

    Returned if the e-mail is successfully inserted in the database. The return code just repeats the HTTP status code.

  • 404 - Failed [expand]

    The response body contain 'application/json' (property)

    Example
    {
  "support_data":{ 
    "feed": {
      "title":"support data", 
      "entries":{
        "code":404,
        "message":"Failed to insert e-mail in database"
      }
    }
  },
  "responseDetails":null,
  "responseStatus":404
}

    Returned if the insert operation failed to insert new e-mail in the database. The text brings the details about the details of the problem.

  • 503 - Service Temporarily Unavailable [expand]

    The application server or reverse proxy are not available. One of them or both might crash.

Data Read Methods

projects

Retrieves the list of project and project owners. The short names are then used in other commands to fetch the case details.

request query parameters
parameter value description

integer

Determines whether the retrieved list contain all oder active projects only.

search

GET, POST

/motcs/rest?cmd=search&data={parameters}

Retrieves the list of support cases fitting the search pattern. The returned structure contain all the available information linked with the case in the ITSM Database. Search is using the SQL clause LIKE %pattern% to identify all the cases that fit the search criteria. The search command is searching the caseNo. only.

request query parameters
parameter value description

pattern

string

The support case number of just fragment of the support case number.

Following example is searching for all cases that contain the fragment 1642. It will return the list of all cases that contain the fragment in any position. E.g. the fragment 1642 will be found in all those numbers: 1642, 16423, 81642, 816423.

Example

http://host/motcs/rest?cmd=search&data={"pattern":"1642"}
available response representations:
  • 200 - OK [expand]

    The response body contain 'application/json' (property)

    Example
    
{
   "support_data": {
      "feed": {
         "title": "support data",
         "entries": [
            {
               "id": 197,
               "case": "1642672",
               "description": "RE: OTCS Ticket 1642672 : Verschl?sselte DocIDs - Exchange Archive",
               "status": "Bug-Open Ticket",
               "details": "22.2.2013 - Niels creates the idea algorithm and Michelle needs to start ECCN form to change the number.<br/>",
               "jira": null,
               "icon": "resources/images/iQuestion.png",
               "leaf": "true",
               "patches": "Patch: "
            }
         ]
      }
   },
   "responseDetails": null,
   "responseStatus": 200
}

    Returned if the query request finds one record only. The list might contain more entries when the searched pattern fits more support case numbers.

  • 404 - Failed [expand]

    The response body contain 'application/json' (property)

    Example

  "support_data":{ 
    "feed": {
      "title":"support data", 
      "entries":{
        "code":404,
        "message":"Search failed"
